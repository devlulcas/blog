# COISAS QUE QUERO APRENDER

Neste ano eu estive trabalhando com PHP, Javascript e Typescript. Tem sido uma ótima oportunidade e eu já aprendi muita coisa.

> Entrei sabendo _quase nada_ de PHP e Typescript e agora me sinto bem confortável com a sintaxe. Isso não quer dizer que eu seja um mestre, só quer dizer que eu não sinto dor ao escrever o código no dia a dia.

Meu maior _ponto de dor_ tem sido o SQL e lidar com o banco de dados no geral.

Quando lido com um banco simples é tudo muito bom e tranquilo, quando lido com um banco mais complexo eu sofro e me sinto sufocado.

Talvez essa seja uma sensação mais normal, mas eu gostaria de passar menos sufoco.

**Aí entra a primeira coisa que eu quero aprender:**

## SQL ![POSTGRESQL](https://img.shields.io/static/v1?label=POSTGRE&labelColor=5760cc&message=SQL&color=000000&logo=POSTGRESQL&logoColor=ffffff&style=flat-square) ![SQL](https://img.shields.io/static/v1?label=FIREBIRD&labelColor=f9c400&message=SQL&color=000000&logo=SQL&logoColor=ffffff&style=flat-square)

Pretendo aprender mais sobre SQL, construir queries mais performáticas e entender melhor o que está acontecendo por trás das operações.

No trabalho usamos o **Firebird**, um SGBD bem popular entre a galera que programa em Pascal, especialmente com quem usa Lazarus.

> Eu acho o **Firebird** um banco interessante, mas não é o mais atraente de todos em questão de comunidade e popularidade. Mesmo assim eu pretendo aprender mais sobre ele, afinal eu tenho que saber usá-lo para resolver os problemas do dia a dia.

## TYPESCRIPT ![TYPESCRIPT](https://img.shields.io/static/v1?label=TYPESCRIPT&labelColor=3589c1&message=JS&color=000000&logo=TYPESCRIPT&logoColor=ffffff&style=flat-square) ![JAVASCRIPT](https://img.shields.io/static/v1?label=JAVASCRIPT&labelColor=f43314&message=JS&color=000000&logo=JAVASCRIPT&logoColor=ffffff&style=flat-square)

Apesar de já me sentir confortável com o **Javascript**, eu sei que ainda não sei tanto quanto poderia. Quero entender bem como tudo funciona por trás das cortinas mágicas dos frameworks, não quer dizer que eu queira saber todos os detalhes, mas um conhecimento parcial ajuda bastante.

Já em relação ao **Typescript** eu ainda preciso melhorar bastante. A linguagem é ótima e ter um sistema de tipagem ajuda a prevenir erros bobos e deixa o código naturalmente mais documentado.

Durante o tempo que fiquei procurando boas razões para aprender Typescript vi também que o sistema de tipos ajuda até a reduzir a quantidade de testes que devem ser escritos.

Para mim o problema em aprender o Typescript não é bem a linguagem em sí, mas sim as bibliotecas e frameworks com o qual usamos ela. Não vejo uma situação no meu dia a dia na qual eu vá usar o Typescript por si só, sempre estou usando com um framework como o Svelte ou NextJS.

## REACT E NEXTJS

Como eu tinha dito, o problema não é bem o TS...

Aprender React tem sido razoavelmente prazeroso. A biblioteca possui alguns traços do paradigma funcional, é bem próxima do Javascript na forma de escrever e é performática o suficiente.

Na minha opinião ainda é mais confortável escrever Svelte, mas considerando o momento atual do mercado o React ainda é bem forte e vale muito a pena aprender. O Svelte tem um gostinho de um HTML empoderado com Javascript, um framework completinho, também considero muito bom aprender.

Já tenho alguns projetos planejados para usar o React e Svelte. Meu desejo é começar a ignorar os zilhões de frameworks que estão surgindo e focar principalmente no React, mas vou manter um olhinho disponível para o Svelte e Solid.

Quando crio projetos no qual quero usar Svelte eu utilizo o Svelte Kit por conta do Server Side Rendering, roteamento via sistema de arquivos, rotas de API e performance deverás agradável. Quando considero isso vejo que vale a pena criar esses novos projetos em React utilizando o framework Next JS.

Enfim, estou interessado em frameworks que consigam gerar o HTML, CSS e JS no servidor, diminuindo o Javascript entregado ao cliente, mas que ainda nos permitem utilizar as ferramentas modernas do ecossistema web (frontend).

## ELIXIR ![ELIXIR](https://img.shields.io/static/v1?label=ELIXIR&labelColor=5f14ff&message=LANG&color=000000&logo=ELIXIR&logoColor=ffffff&style=flat-square) ![elixir](https://img.shields.io/static/v1?label=PHOENIX&labelColor=5f14ff&message=FRAMEWORK&color=000000&logo=elixir&logoColor=feb000&style=flat-square)

Elixir é interessante por:

- ser uma linguagem de programação funcional
- possuir sintaxe próxima a Ruby/Python (bem bonitinha)
- ser moderna
- ser brasileira
- possuir características que facilitam o paralelismo
- ser utilizada em grandes empresas
- possuir o ecossistema da BEAM por trás
- ser compilada para byte code
- possuir um framework full stack como o Rails do Ruby
- possuir comunidade interessante e ativa
- possibilidade de escrever testes nativamente de forma moderna e simples
- um gerenciador de pacotes bem evoluído
- _logo legal_

Além dessas questões ainda existem projetos como o Livebook e Liveview que me deixam bem curioso.

Penso em aprender Elixir principalmente para entender melhor o paradigma funcional, sem pular para algo que aparenta ser mais complexo como Haskell, Lisp ou Clojure.

Outra coisa interessante no Elixir é que existem boas vagas que pedem conhecimento na linguagem.

## GO ![GO](https://img.shields.io/static/v1?label=GO&labelColor=14ffd8&message=LANG&color=000000&logo=GO&logoColor=000000&style=flat-square)

Go é interessante por:

- ser uma linguagem de programação multi-paradigma, mas mais voltada para algo procedural/funcional (até onde eu entendi).
- possuir sintaxe absurdamente simples (não tão bonitinha, mas talvez minha opinião mude)
- ser moderna
- possuir características que facilitam o paralelismo
- ser utilizada em grandes empresas (bem grandes meeeeesmo)
- ser compilada para código de máquina diretamente
- ser MUITO rápida
- habilitar a produtividade em pouco tempo
- ser boa para criar aplicações WEB
- possuir comunidade interessante e ativa
- criar binários auto-contidos
- _mascote fofo_

GO pode ser uma boa linguagem para aproveitar vagas internacionais e para ter uma linguagem verdadeiramente rápida e eficaz na caixa de ferramentas.

Até agora a única coisa que tem me desmotivado com o GO é que ainda não engoli direito a sintaxe.

## RUST ![RUST](https://img.shields.io/static/v1?label=RUST&labelColor=ffa714&message=LANG&color=000000&logo=RUST&logoColor=000000&style=flat-square)

Rust é interessante por:

- ser uma linguagem de programação multi-paradigma, mas com um gostinho mais funcional
- possuir sintaxe bem moderna (bonitinha, porém depende... vi uns códigos sinistros)
- ser moderna
- ser EXTREMAMENTE rápida
- possuir características que facilitam o paralelismo
- ser utilizada em grandes empresas
- possuir erros ótimos
- prevenir erros e falhas do programador
- não possuir garbage colector
- ser baixo nível
- ser compilada para código de máquina
- possuir comunidade interessante e ativa
- ser muito utilizada no ecossistema JS
- gerar binários auto-contidos
- **[isso](https://youtu.be/LDU_Txk06tM?t=73)**

Existem projetos interessantes no mundo Rust que me empolgam.

Ter a possibilidade de escrever aplicações desktop performáticas de verdade também é muito legal.

Tenho pensado seriamente no Rust, pois seria um ótimo complemento para meus projetos frontend e abriria minha mente para novas formas de pensar.

Rust também parece uma boa ferramenta para construir aplicações de linha de comando, algo que eu sinto falta de fazer.

## Finalizando

Por hora vou focar no que preciso para o dia a dia.

Typescript, Javascript, React, Next e SQL. Sendo que os primeiros quatro entram num grupinho só que é de ferramentas frontend.

Quando puder eu verei um pouco de Elixir para escrever ao menos um projeto com a linguagem.

Futuramente farei o mesmo com Go e Rust. Veremos depois o que vai me prender mais e me fazer ter vontade de criar mais projetos.
